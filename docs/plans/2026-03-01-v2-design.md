# Controller v2 — Design Document
*2026-03-01*

## Overview

v2 replaces raw YAML editing with a form-based approach. Templates are now structured records (network config + selected services). YAML is generated at deploy time rather than stored as a file. A new Services page manages Docker service configurations independently of templates.

Core philosophy: **companion app, not a replacement for Docker Desktop.** Controller makes docker-compose files invisible to the user — not by limiting what's possible, but by presenting the most common patterns as simple form fields.

---

## Navigation & Layout

### Dashboard (default view)
- Left sidebar removed entirely
- Top bar: logo + "Controller" | CPU% | Memory% | Disk% | Running:N | gear icon (top right)
- Service card grid + deploy/stop/pull action bar — unchanged from v1
- Gear icon → Settings page

### Settings page
- Back arrow (top left) returns to Dashboard
- Two tabs: **Templates** (default) | **Services**

### Onboarding (shown when no templates exist)
- "Get Started" button → creates empty template → opens Settings → Templates tab
- YAML drag-and-drop upload removed (v2 builds templates via form; import can be added later)

## Settings: Templates Tab

### Layout
Two-panel: left list + right detail form.

**Left panel**
- Scrollable list of all templates
- "New Template" button → dialog: *"Template name: ________"* → creates record → opens detail

**Right panel — template detail**

Header: template name field (editable inline) | `</>` button (top right)

**Section 1 — Network**

| Field | Type | Default | Notes |
|---|---|---|---|
| Name | text input | `appnet` | ⓘ tooltip (see below) |
| Type | radio buttons | Local (bridge) | Three options |

Network name tooltip:
> *"All services on the same network can reach each other. For personal use, one network is fine. For production, consider isolating unrelated services into separate networks."*

Network type options:
- **Local (bridge)** — default. *"Creates an isolated network on this machine."*
- **Block external internet (internal)** — sets `internal: true`. Local is auto-checked alongside this.
- **Join a shared network (external)** — sets `external: true`. Reveals text input for existing network name. Disables the other two options.

Radio interaction rules:
- Selecting *external* disables Local and Internal
- Selecting *internal* keeps Local checked alongside it
- Selecting *local* alone clears Internal

**Section 2 — Services**

*"Select the services to include in this template:"*
- Checkbox list of all available services (image present locally + not marked Unavailable)
- Previously-selected services that are now unavailable show greyed with note: *"ollama was removed (image not found)"*

**Footer**: `[Delete Template]` — — — `[Save Template]`

**`</>` modal**
- Read-only generated YAML
- Copy button
- Title: *"Generated YAML — {template name}"*
- No editing

## Settings: Services Tab

### Layout
Two-panel: left list + right detail form.

**Left panel**
- Scrollable list of all service records
- Each row: service name, image tag, `image not found` badge if image missing from Docker
- "Add Service" button at top

**Right panel — service detail**

| Field | Type | Notes |
|---|---|---|
| Service name | text | Compose key, e.g. `ollama` |
| Image | text | e.g. `ollama/ollama:latest` |
| Ports | list editor | Add/remove rows, e.g. `11434:11434` |
| Volumes | list editor | e.g. `ollama:/root/.ollama` |
| Environment variables | list editor | Key=value rows |
| Restart policy | dropdown | `unless-stopped` / `always` / `on-failure` / `no` |
| Unavailable to templates | toggle | Label: *"Hide this service from template configuration"* |

**Footer**: `[Remove from Controller]` — — — `[Save Service]`

Remove confirmation dialog:
> *"This removes [name] from Controller's service list. Your port and volume configuration will be lost.*
> *To also delete the Docker image, run: `docker image rm {image}` or use Docker Desktop → Images."*
> Buttons: **Remove from Controller** | **Cancel**

### "Add Service" dialog
- Image name input (e.g. `ghcr.io/open-webui/open-webui:main`)
- Below input: list of locally available Docker images not yet in the services list (click to populate)
- **"Add to Services list"** button — creates record, opens detail form
  - Small note: *"Image will be pulled automatically on first deploy if not available locally."*
- **"Add & Pull"** button — runs `docker pull` first, then creates record

### Image sync
- Status poll (every 5 seconds) compares service records against `docker images` output
- Services whose image is not present get `image not found` badge automatically
- These services are also hidden from template service checklists (same as Unavailable)

## Service Unavailability Rules

Two reasons a service won't appear in a template's service checklist:

| Reason | Source | Message in Services list |
|---|---|---|
| User toggled Unavailable | stored in `services.json` | *(no badge — intentional)* |
| Image not found locally | computed from `docker images` | `image not found` badge |

Both are silent in the template checklist — the service simply doesn't appear.
Service JSON records are **never auto-deleted** when an image is removed from Docker (config is preserved). User must explicitly "Remove from Controller" to clean up stale records.

## YAML Generation

YAML is generated at deploy time and for `</>` preview. No YAML files stored on disk.

Generated structure:
```yaml
networks:
  {network.name}:
    driver: bridge          # always present unless external
    internal: true          # if network.internal
    external: true          # if network.external
    name: {externalName}    # if network.external

volumes:
  {volume_name}:            # one entry per unique volume across all selected services

services:
  {service.name}:
    image: {service.image}
    ports: [...]
    volumes: [...]
    environment: [...]
    restart: {service.restart}
    networks:
      - {network.name}
```

Volumes are plain named volumes (no `external: true`) — Docker creates them on first deploy and reuses them on every subsequent deploy.

## Data Model

```
data/
  settings.json     # { publishDir, activeTemplateId, lastPulledAll }
  templates.json    # [{ id, name, createdAt, lastPulled,
                    #    network: { name, driver, internal, external, externalName },
                    #    serviceIds: [] }]
  services.json     # [{ id, name, image, ports: [], volumes: [],
                    #    environment: {}, restart, unavailable }]
```

No more `data/templates/{id}/docker-compose.yml` — removed in v2.

## API Changes

### New endpoints (Services)
```
GET    /api/services              # list all + image-present status (from docker images)
POST   /api/services              # create service record
PUT    /api/services/{id}         # update service config
DELETE /api/services/{id}         # remove from Controller (not from Docker)
POST   /api/services/{id}/pull    # docker pull {image}
```

### Changed endpoints (Templates)
```
PUT  /api/templates/{id}          # now updates name + network config + serviceIds
GET  /api/templates/{id}/compose  # generates + returns YAML (was: read raw file)
```

### Removed endpoints
```
POST /api/templates               # was: upload YAML file — replaced by form-based creation
PUT  /api/templates/{id}/compose  # no longer needed (no raw YAML editing)
```

### Unchanged endpoints
```
GET  /api/status
DELETE /api/templates/{id}
POST /api/templates/{id}/deploy
POST /api/templates/{id}/stop
POST /api/templates/{id}/pull
POST /api/pull-all
POST /api/containers/{name}/start
POST /api/containers/{name}/stop
GET  /api/settings
PUT  /api/settings
```
